# 000: 과거 발생 오류와 해결 방법 (누적 기록)

CLAUDE.md를 멀티 에이전트 워크플로우로 개편하기 이전까지 누적된 오류/해결 기록.

## CORS

- `credentials: true`와 `origin: '*'` 조합은 브라우저가 거부함.
  → `CORS_ORIGIN='*'`인 경우 요청의 Origin을 그대로 반사(reflect)하는 함수로 처리할 것.

## DB 스키마

- `settings` 테이블의 `key`/`value` 컬럼명은 MySQL 예약어라 Railway MySQL Query 탭 등에서 백틱이 셸로 해석되는 문제 발생.
  → `setting_key`/`setting_val`처럼 예약어를 피한 컬럼명 사용.
- `seed.sql`의 테스트 유저 비밀번호 해시가 실제 비밀번호(`test1234`)와 불일치했던 적 있음.
  → 시드용 bcrypt 해시 추가/변경 시 반드시 해당 평문 비밀번호로 재생성한 해시인지 검증.

## DB 마이그레이션 (sql/migrations, deploy.sh)

- 새 마이그레이션 파일 번호가 기존 번호와 충돌하면 배포 실패.
  → 새 마이그레이션 추가 시 `sql/migrations/`의 기존 번호를 확인하고 다음 번호 사용.
- `schema.sql`에 이미 포함된 내용을 별도 마이그레이션 파일로 또 만들면, `schema_migrations` 테이블 첫 생성 시 항상 충돌.
  → `schema.sql`에 이미 반영된 변경은 별도 마이그레이션으로 만들지 말거나, `schema_migrations` 테이블 최초 생성 시 "이미 적용됨"으로 자동 기록.

## 배포 (Railway / nixpacks / Oracle Cloud)

- 모노레포 구조(루트에 package.json 없음)에서는 Nixpacks가 Node.js를 자동 감지하지 못함.
  → `nixpacks.toml`에 `providers = ["node"]` 명시 (패키지명을 `nodejs_22` 등으로 직접 지정하면 오류 발생, provider 방식 사용).
- 현재는 Vercel/Railway 설정 대신 Oracle Cloud + `deploy.sh`(git pull, DB 마이그레이션, 프론트 빌드, pm2 재시작) 방식 사용.

## API / 프론트엔드 연동

- 투표가 없는 게시글의 폴(poll) 조회 API가 404를 반환하면 콘솔에 불필요한 에러 로그가 남음.
  → 데이터가 "없음"이 정상 상태인 리소스는 404 대신 200 + `null` 등으로 응답하고, 프론트엔드에서 404 분기 처리를 두지 않는다.
